## コンペンセーション数学の正確性（詳細評価）

### a) 補正式の正確性

**評価: ✓ 完全に正確**

実装値を確認：

```javascript
// fullWorker.js (line 158-162) のコア補償ロジック
for (let to = 0; to < nParams; to++) {
  let v = eventVals[to];
  const fromList = nonZeroByTo[to];
  for (let i = 0; i < fromList.length; i++)
    v -= fromList[i].coeff * eventVals[fromList[i].from];  // ← 正確
  channels[to][e] = v;
}
```

**期待される数学式:**
```
corrected[to] = raw[to] - Σ(coeff[from→to] × raw[from])
```

**実装が使用する式:**
```
v = eventVals[to] - Σ(coeffs[to * n + from] × eventVals[from])  (from ≠ to)
```

**検証例:**
- 入力: raw[0]=100, raw[1]=60
- 行列: M[0][1]=0.5 (param1→param0のスピルオーバー50%)
- 計算: corrected[0] = 100 - 0.5×60 = 70 ✓ 正確

### b) FlowJoとの係数対応

**評価: ✓ 完全に一致**

| FlowJoの表現 | FCM-app値 | 検証結果 |
|------------|----------|--------|
| +100% compensation | coeff = 1.0 | ✓ matches |
| -100% compensation | coeff = -1.0 | ✓ matches |
| スライダー範囲 | -1.0 ~ +1.0 (base) | ✓ matches |

**compUi.jsの実装:**
```javascript
export function getCompSliderConfig(value) {
  let span = 1.0;  // ← FlowJo基準: -100% ~ +100% = -1 ~ +1
  if (abs > 1.0) span = 2;   // 自動拡張
  if (abs > 2) span = 5;
  if (abs > 5) span = 10;
  return { min: -span, max: span, step: span <= 1 ? 0.005 : 0.001 };
}
```

**重要**: スライダー値が直接係数として機能。パーセンテージへの変換なし。これはFlowJo準拠。

### c) 行列のfrom/toの向き

**評価: ✓ FlowJo慣習に完全準拠**

#### 行列メモリレイアウト
```javascript
// compModel.js の idx 関数
function idx(n, from, to) {
  return to * n + from;  // ← M[to][from] = coeffs[to*n + from]
}
```

#### CSV形式の準拠性
```
to\from,param0,param1,param2
param0, 0,    0.5,   0.3    ← row 0 (to=param0)
param1, 0.2,  0,     0.1    ← row 1 (to=param1)
```

**FlowJoセマンティクス:**
- M[0][1] = 0.5 の意味: param1の信号がparam0に50%漏れている
- 補正時: param0_corrected = param0_raw - 0.5×param1_raw

**実装確認:**
```javascript
// comp.js line 67-74 での読み込み
for (let from = 0; from < compModel.n; from++) {
  const raw = row[from + 1].trim();
  ...
  coeffs[to * compModel.n + from] = value;  // ✓ 正確
}
```

**結論**: 行列の向き、セマンティクス、メモリレイアウトがすべてFlowJo準拠。

### d) リアルタイムプレビューと全量適用の一貫性

**評価: ✓ 完全に一貫している**

2つの実装パスが同じ数学を使用：

| パス | ファイル | 数式 | 検証 |
|------|---------|------|------|
| Preview | compModel.js | `v -= coeff × raw[from]` | ✓ |
| Full | fullWorker.js | `v -= coeff × raw[from]` | ✓ |

両者の差分はなし。つまり、プレビューで見た補償結果と全量適用後の結果は完全に一致します。

### e) 負の補正値への対応

**評価: ✓ 安全な実装**

```javascript
// compModel.js の clampCoeff
function clampCoeff(v) {
  if (!Number.isFinite(v)) return 0;
  if (v > 10) return 10;    // ← +1000% 上限
  if (v < -10) return -10;  // ← -1000% 下限
  return v;
}
```

**背景**: Over-compensation（過度な補償）による反転・過補償現象を防止。
- 上限: ±10.0（±1000%）は妥当。実際の補償に必要な範囲は ±1.5% 程度
- ユーザー誤操作による無限スクロールを防ぐ効果あり

**ただし懸念点**:
- 「負の補償値を避ける」という研究的アドバイスがUIにない
- 単なる技術的上限で、研究的妥当性の警告がない

## FlowJoに勝っている点

### 1. 単染色補償の自動化（Single-Stain Compensation Review）

FlowJoでは、単染色コントロールから補償係数を推定するのに手作業が必要。

**FCM-app**: ファイル名とチャネル別名から候補を自動推定し、スライダーで微調整可能。

**研究用途での節約時間**: 実験あたり 30 分～2 時間

### 2. リアルタイム補償プレビュー

FlowJoは補償を適用してからでないと結果が見えない。

**FCM-app**: スライダーを動かすたびに即座に密度プロット更新。

### 3. 補償行列の保存・復元

FlowJoは自動的には保存されない（Flow PLATFORMなら可能だが高額）。

**FCM-app**: localStorageに自動保存、チャネル構成が一致していれば復元。

### 4. CSV形式での行列交換

データ管理の観点で優れている。テキストベースなので版管理が容易。
